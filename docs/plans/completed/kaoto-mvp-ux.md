# Kaoto MVP: пользовательский опыт

Производный документ к [kaoto-mvp-finalization.md](./kaoto-mvp-finalization.md). Описывает MVP **с точки зрения пользователя**: что он сможет делать после деплоя, что — нет, и в чём разница между двумя путями добавления шагов (нашей палитрой и родным механизмом Kaoto).

Контекст инжиниринга — в [kaoto-mvp-finalization.md](./kaoto-mvp-finalization.md) и [kaoto-integration-plan.md](../kaoto-integration-plan.md).

## Что получит пользователь

### Открытие и навигация

- Чистая страница редактора Camel DSL: канвас на всю ширину по умолчанию, header в 2 строки (title + name + state + Save / recordRef + переключатель режимов).
- Три режима просмотра: `Visual` (только канвас), `Split` (60/40 канвас + Monaco), `YAML` (только Monaco). Дефолт — `Visual`. Состояние YAML не теряется при переключении.

### Редактирование существующего маршрута

- Один способ редактировать шаг — клик на ноду в канвасе → встроенная Kaoto-форма `CanvasSideBar` с табами Required/All/Modified. Двух разных форм для одного шага больше нет.
- Citeck-шаги (`ecos-event:`, `ecos-endpoint:`) рендерятся той же формой, что и стандартные Camel-компоненты — пользователь не различает «наши» и «чужие» шаги, UX единый.
- Двусторонняя синхронизация канвас ↔ YAML: канвас → Monaco live, Monaco → канвас по кнопке `Apply to canvas` (manual, чтобы не падать на промежуточно-невалидном YAML).

### Добавление шагов — три пути, ни одной дыры

- Кнопка `+ Add step` над канвасом → modal с 18 курированными тайлами в трёх категориях (10 DSL-core: setHeader/setBody/choice/filter/split/log/marshal/…; 4 Citeck-domain: ecos-event/ecos-endpoint/records-dao/get-record-atts; 4 External: rabbitmq/http/smtp/direct). Click-to-add → шаг появляется в конце `steps:` и сразу рисуется на канвасе (без `Apply`).
- Полный каталог Camel (~613 компонентов) доступен через родной Kaoto-«+ Add step» на любой ноде.
- Перемещение шагов — drag-reorder по канвасу (родной Kaoto).

Подробное сравнение двух механизмов добавления — см. [секцию ниже](#два-пути-добавления-шагов--в-чём-разница).

### Создание маршрута с нуля

- URL `?new=true` открывает editor в new-mode: пустой канвас, Trigger-dropdown в header'е (5 Citeck-event триггеров + 3 Camel-core: timer/direct/quartz). Save disabled пока триггер не выбран.
- Pick trigger → канвас оживает с одной `from`-нодой → пользователь добавляет шаги через палитру → Save → запись создаётся в `integrations/camel-dsl@`, URL обновляется на `?recordRef=...`, дальнейшие сейвы идут по update-path. Reload по новому URL — обычный edit-flow.

### Косметика

- Скрыт лишний RuntimeSelector dropdown («Camel Main 4.14.2.redhat-00011») — runtime у нас зафиксирован, переключать нечего.
- Полезные кнопки `ContextToolbar` Kaoto (fit-to-screen, layout-direction) остаются.

## Что пользователь НЕ получит

### Палитра

- Нет drag'а тайла из палитры на конкретную точку канваса одним жестом — только click-to-add в конец + reorder drag'ом по канвасу.
- Нет «вставить после выбранного шага» из нашей палитры (Kaoto не экспортирует selection API наружу). Workaround — родной «+ Add step» на ноде.
- Нет полного каталога 613 компонентов в нашей сетке — только 18 курированных тайлов. Остальное — через родной механизм Kaoto.

### Выбор и контекст

- Нет внешнего outline/дерева шагов рядом с канвасом — единственный способ выбрать шаг это клик по ноде в канвасе.
- При добавлении шага из палитры текущий выбор в `CanvasSideBar` сбрасывается (sidebar закрывается) — auto-remount канваса. Пользователь после добавления шага всегда «начинает с чистого листа выбора». Может уйти, если мини-spike покажет, что remount можно избежать.
- Нет двусторонней связи selection ↔ outline — outline'а просто нет.

### Создание маршрута

- В new-mode нет wizard'а первого шага: после выбора триггера — пустой `steps:`, дальше пользователь сам.
- Нет шаблонов маршрутов (`Notification on status change`, `Sync with X` и т.п.) — только bare-trigger.
- Нет валидации триггера до создания (например, проверка существования `recordType` в emodel).
- Нет cancel-confirm modal'а при выходе из new-mode без сохранения — полагаемся на browser'ный `beforeunload`.
- **Нет entry-point в журнале «Создать → Camel DSL → новый маршрут»** в рамках этого MVP — открыть `?new=true` можно только по прямой ссылке/закладке. createVariant заводится отдельным PR в `ecos-integrations`, может ехать параллельно или после.

### Формы и подсказки

- Нет автокомплита `{{ecos-secret:<id>/<field>}}` в parameter-fields — `username`/`password`/`token` пользователь печатает руками. Будет отдельной задачей через `SuggestionRegistryProvider` (≈0.5 дня после MVP).
- `bean:ecos-secret/<id>` как отдельная нода в палитре отсутствует — это namespace property-функции, а не шаг (раньше был ошибочный override, в MVP убирается).

### Управление маршрутом

- Старт/стоп/suspend контекста — не в нашем UI, а через готовое UI-action `start-camel-dsl` из типа (отдельная кнопка в журнале/карточке).
- Copy шага через UI нет — только через YAML-таб.
- Undo/redo на уровне страницы нет (есть только undo внутри Monaco для текстовых правок).
- Конфликт сохранений (двое правят одну запись) не обрабатывается — last-write-wins.
- Disabled-Save с tooltip'ом «No permission» — есть, но это всё; нет более тонкой фильтрации полей по правам.

### Локализация и адаптив

- UI Kaoto и наши строки — только English. i18n — отдельная итерация.
- Mobile/планшет не поддерживается, минимум 1024×600.

### Архитектурные ограничения (под капотом, но влияют на пользователя)

- Embedding через `RouteVisualization` — undocumented public surface Kaoto. На минор-bump'ах upstream'а возможны мелкие регрессии, которые ловятся только нашими тестами. Альтернатива (iframe + `multiplying-architecture`) — это 5–10 дней миграции, отложено за горизонт MVP с check-in'ом T+2 месяца.

### Резюме одной фразой

Пользователь получает функционально полный редактор для типового потока «открыл маршрут → отредактировал шаги через одну форму → добавил новые из палитры или родного каталога → сохранил», и базовый сценарий «создать маршрут с нуля по прямой ссылке». Всё, что выходит за рамки типового — drag-from-palette, шаблоны, autocomplete секретов, journal entry-point, multi-route, i18n, mobile — остаётся за пределами MVP с явными follow-up'ами.

## Два пути добавления шагов — в чём разница

В MVP **намеренно сосуществуют** два механизма. Они отличаются по четырём осям: где жмёшь, что видишь, куда вставится, насколько широкий выбор.

### Наша палитра (`+ Add step` над канвасом)

| Аспект | Что |
|---|---|
| Триггер | Кнопка в нашем тулбаре над канвасом, всегда видна |
| Контекст | Ничего не выбрано (или selection игнорируется) |
| Содержимое | 18 курированных тайлов в трёх категориях: `DSL — Camel core` / `Citeck-domain` / `External integrations` |
| Citeck-шаги | **Выделены в отдельную категорию** (`ecos-event`, `ecos-endpoint`, `records-dao`, `get-record-atts`) — пользователь сразу видит платформенные хелперы |
| Точка вставки | **Всегда конец** `steps:` текущего route'а |
| Перемещение в нужное место | Отдельным жестом — drag по канвасу |
| Snippet | Наш — мы сами решаем дефолтные значения для каждого тайла (placeholder URI, разумные параметры) |
| Покрытие | ~3% каталога (18 / ~613), но это самые частые штуки |
| Discoverability | Высокая для типовых сценариев («что обычно делают в Camel?») |

### Родной Kaoto «+ Add step» на ноде канваса

| Аспект | Что |
|---|---|
| Триггер | Маленькая кнопка `+`, появляется **на каждой ноде канваса** при hover/select |
| Контекст | Привязан к конкретной ноде — это ключевое отличие |
| Содержимое | **Все ~613 компонентов Camel-каталога** одной плоской сеткой, без наших категорий |
| Citeck-шаги | Растворены среди 613 других, ищутся текстовым поиском по слову `ecos-` (если пользователь знает что искать) |
| Точка вставки | **После выбранной ноды** — там, где жал `+` |
| Перемещение в нужное место | Не нужно — сразу в нужном месте |
| Snippet | Дефолтный от Kaoto/Camel-каталога, без Citeck-tuning'а |
| Покрытие | Полное |
| Discoverability | Низкая — пользователю нужно знать имя компонента или активно фильтровать |

### Где они дополняют друг друга

"Хочу типовой шаг (log/setHeader/choice/...) или Citeck-          |  наша палитра
 интеграцию (ecos-event, ecos-endpoint), не помню точный URI"    |
                                                                  |

                                                                  |

                                                                  |
"Создаю route с нуля, ещё нет ни одной ноды"                      |  только наша палитра

### Главный UX-зазор

Наша палитра не умеет **insert-after-selected**. Хочешь типовой `log` между двумя шагами? Два жеста:
1. Click `+ Add step` → click `log` тайл → шаг улетает в конец.
2. Drag-reorder этой ноды на нужную позицию по канвасу.

Или один жест через родной Kaoto:
1. Click `+` на ноде, после которой нужен `log` → выбрать `log` из 613-сетки.

Первый путь приятнее по «визуальному поиску» (категории, иконки, известные шаги), второй — по позиционированию. В MVP они сосуществуют именно потому, что каждый покрывает то, что плохо умеет другой. Insert-after-selected в нашей палитре закроет gap и схлопнет два механизма в один, но это требует Kaoto API селекшена, которого наружу нет (см. §«Что НЕ входит» в [kaoto-mvp-finalization.md](./kaoto-mvp-finalization.md)).

### Когда ждать схлопывания в один путь

Когда сбудется хотя бы один из двух:

- Kaoto экспортирует `useEntityContext`/selection API в `public-api.ts` (сейчас он в `lib/`, internal-only) — наша палитра научится insert-after-selected.
- Мигрируем embedding на iframe/`multiplying-architecture` — там родная палитра-sidebar поверх всех 613 компонентов с insert-в-точку — наша своя становится ненужной.

Оба варианта — post-MVP, в [kaoto-mvp-finalization.md §6.5](./kaoto-mvp-finalization.md) зафиксирован check-in T+2 месяца на пересмотр обоих триггеров.
